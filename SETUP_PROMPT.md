# セットアップ用プロンプト（コピペ用）

以下を AI エージェントに渡してください。

```text
新規プロジェクトの初期セットアップをお願いします。

前提:
- このリポジトリは開発プロジェクト向けの docs-first テンプレートです。
- まず `docs/initial-brief.md` を読み、要件の整理と不足情報の洗い出しを優先してください。
- この時点では完成済みの実装は不要です。必要な文書を具体化し、実装開始の判断材料を揃えることを重視してください。
- まずプロジェクト種別を整理してください（例: Web アプリ、CLI、ライブラリ、バッチ、社内ツール）。
- 実装スタックは、要件に最も近い最小構成を提案してください。Web アプリの場合は `Bun + Vite + TypeScript` も候補です。
- 不足情報があれば質問するか、仮置きした前提を明示してください。

やってほしいこと:
1. `docs/initial-brief.md` の記述をもとに、不明点・前提・判断保留事項を整理
2. 実プロジェクト用の `README.md` を具体化し、以下を反映
   - Project Name: <PROJECT_NAME>
   - 概要: <PROJECT_SUMMARY>
   - 目的: <PROJECT_GOAL>
   - 対象範囲: <PROJECT_SCOPE>
   - 非目的: <NON_GOALS>
3. `docs/project-spec.md` を今回のプロジェクト向けに更新（目的、非目的、非機能要件、受け入れ条件、プロジェクト種別、技術方針を明確化）
4. `AGENTS.md` を今回のプロジェクト向けに更新（目的・維持対象・命名規則・実装時の標準コマンドを明確化）
5. `docs/coding-rules.md` は既存ルールをベースに、採用スタックに必要なものだけ最小限で追記
6. `.gitignore` を作成/更新し、最低限以下を含める
   - node_modules/
   - dist/
   - coverage/
   - .env
   - *.pem
   - *.key
7. 必要なら `CONTRIBUTING.md` と `CHANGELOG.md` も更新
8. 実装雛形を作るべきかどうかを判断し、必要な場合のみ最小構成で提案または実装
9. 変更後に、判断理由・変更ファイル一覧・未確定事項を報告

制約:
- 変更は最小差分で。
- 推測で重い構成を入れない（必要なものだけ）。
- 文書だけで十分な段階なら、不要な実装ファイルは作らない。
- コミットはしない。
```

## 置換用パラメータ例
- `<PROJECT_NAME>`: Example App
- `<PROJECT_SUMMARY>`: 社内向けタスク管理ツール
- `<PROJECT_GOAL>`: タスク登録と進捗可視化を簡単にする
- `<PROJECT_SCOPE>`: 社内メンバーがブラウザから利用する
- `<NON_GOALS>`: モバイルアプリ対応、外部課金連携
