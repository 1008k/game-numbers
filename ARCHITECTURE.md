# Architecture Overview

この文書は、コピー後プロジェクトの実装構造を素早く把握する補助文書です。仕様の正本は `docs/project-spec.md`、実装判断は `docs/rules-coding.md` を参照します。

テンプレート段階では、特定の技術スタックやディレクトリ構成を固定しません。実装が始まったら、実態に合わせて更新します。

## 1. 位置づけ
- 対象: コピー後プロジェクトの実装構造
- 目的: 主要モジュール、責務分割、依存関係、外部連携、ビルド/テスト方法を把握しやすくする
- 更新タイミング: ディレクトリ構成、主要モジュール、外部連携、保存形式、ビルド手順が変わったとき

## 2. 実装構造
実装開始後に、主要ディレクトリと役割を記載します。

```text
src/
tests/
docs/
```

## 3. 主要モジュール
実装開始後に、主要モジュール、画面、サービス、ライブラリの責務を記載します。

- `src/`: アプリケーション本体。
- `tests/`: 自動テスト。
- `docs/`: 仕様、設計、運用文書。

## 4. 責務分割
- ドメインロジック、UI、外部連携、永続化を必要以上に結合させない。
- 外部サービスやプラットフォーム差分は、境界となる層に閉じ込める。
- 同じ意味の設定値、判定、feature flagを複数箇所に散在させない。
- 実装が小さい段階では、過剰な抽象化より読みやすさを優先する。

## 5. データ構造
プロジェクト固有の主要データ構造が決まったら記載します。

## 6. 外部連携
外部サービス、API、MCP、認証、決済、分析、通知などを追加する場合は、目的、影響範囲、必要権限、失敗時の扱いを仕様またはtrackに残します。

## 7. ビルド / テスト / デプロイ
- ビルド方法: 実装スタック決定後にREADMEへ追記する。
- テスト方法: 実装スタック決定後にREADMEへ追記する。
- 配布方法: プロジェクト種別に合わせて定義する。
- CI: 必要性と保守負荷を見て導入する。
