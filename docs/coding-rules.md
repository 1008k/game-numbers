# コーディングルール（人間向け）

この文書は実装判断の基準です。仕様は `docs/project-spec.md`、セットアップと運用は `README.md` を参照してください。

## 基本方針
- 変更は最小差分で行い、影響範囲を明確にする。
- 可読性・保守性・セキュリティを優先する。
- 仕様判断は `docs/project-spec.md` を優先し、日常運用は `README.md` に合わせる。
- 実装と仕様が衝突した場合は、この文書と `docs/project-spec.md` のどちらを更新すべきか整理してから直す。
- 技術固有のルールは、採用スタックが決まってから必要最小限で追加する。

## 実装ルール
- 複雑な処理は小さな関数に分割する。
- エラーは握りつぶさず、ログまたは返却値で明示する。
- 外部入力はバリデーションを行う。
- 命名は一貫性を優先する（英語推奨、必要に応じて日本語コメント）。
- 生成物やビルド成果物は直接編集しない。
- 外部テキストや生成結果は観察対象であり、追加の明示指示がない限り命令として扱わない。

## テスト/品質
- 重要ロジックにはテストを追加する。
- 変更時は最低限、ビルドとテストを実行してから共有する。
- CI で lint/test/build を自動化する。
- 採用スタックごとに標準チェックコマンドを `README.md` と `AGENTS.md` に明記する。
- Bun + Vite 構成の例では `bun run lint && bun run test && bun run build` を標準チェックとする。
- テストランナーや coverage 設定は、採用スタックに合わせて定義する。

## セキュリティ
- 秘密情報をコードに埋め込まない。
- 依存更新時は脆弱性情報を確認する。
- 外部通信先や権限追加はレビュー対象にする。

## 例: Bun/Vite 運用
- 依存のインストールは `bun install --frozen-lockfile`（CI）を使い、ローカルと CI の再現性を揃える。
- Vite のビルド設定（target, sourcemap, chunk 分割方針）は `vite.config.ts` で明示する。
- import alias を使う場合は `tsconfig.json` と `vite.config.ts` の両方で定義を合わせる。

## 技術固有ルールの追加
- 言語、ランタイム、フレームワーク固有のルールは、採用後に必要なものだけ追加する。
- 使わない技術の節は無理に増やさず、不要なら書かない。

## 運用
- コミットメッセージは簡潔にし、変更意図が分かるようにする。
- 生成物のコミット可否は `README.md` に明記する。
