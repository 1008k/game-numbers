# 文書表記ルール

この文書は、プロジェクト内の文書表記ルールと例をまとめたものです。優先順位や静的に扱える項目は `docs/policy-index.yaml` を先に参照。実装判断は `docs/rules-coding.md`、仕様は `docs/project-spec.md`、セットアップと運用は `README.md` を参照。

## 基本方針
- 和文中の英単語、英数字、記号の前後には、原則として半角スペースを入れない。
- 例: `AIエージェント`, `Webアプリ`, `Viteがビルドする`, `public配下`
- コマンド、パス、設定キー、識別子、コード断片、複合英語の原表記は、必要に応じてバッククォートで囲んで保持する。
- 例: `bun install --frozen-lockfile`, `service worker`, `vite.config.ts`, `feature flag`
- 可読性のために区切りが必要な場合も、まず語の選び方やバッククォートで解消し、地の文に機械的な空白を増やさない。

## 用語統一
- 同じ対象を指す語は、同一文書内と関連文書間で表記をそろえる。
- 既存文書に正本となる表記がある場合は、それに合わせる。

## 文体
- 文書単位で文体をそろえ、`です・ます` と `である` を混在させない。
- 箇条書きの語尾は、同じ節の中でできるだけそろえる。

## 曖昧さを避ける
- `適宜`, `必要に応じて`, `適切に` などの曖昧な表現は、多用しない。
- 判断を委ねる場合は、判断基準をできるだけ併記する。

## Changelog / release note
- `CHANGELOG.md` や release note は、配布物の動作に関わる利用者向け変更だけを記録する。
- 各項目は1行ずつ簡潔に書き、何が変わったかがすぐ分かる表現にする。
- 内部実装詳細、文書調整、理由説明、実装方針、内部事情は、利用者への影響を理解するために必要な場合を除き含めない。
- 背景や判断理由を補足したい場合は、本文を長くせず `docs/tracks/` の該当項目を参照する。
