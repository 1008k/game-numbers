# プロジェクト仕様書

この文書は設計方針と受け入れ条件をまとめた仕様文書です。日々の実装判断は `docs/rules-coding.md`、セットアップと運用は `README.md` を参照してください。

## この文書の位置づけ
- この文書は「何を作るか」「どこまでを対象にするか」「何を満たせばよいか」を定義する。
- 実装ルールやコーディングスタイルは `docs/rules-coding.md` で管理する。
- 実装と衝突した場合は、この文書と `docs/rules-coding.md` を見直し、必要なら両方を更新する。
- 実装や文書更新の前に、完了条件と確認方法を明確にする。

## 3. 技術方針
- 想定カテゴリ: `<Web app / CLI / Library / Batch / Internal tool / etc.>`
- 実装スタックは、要件に対して最小構成になるものを選ぶ
- Webアプリとして始める場合の例のひとつは `Bun + Vite + TypeScript + Vitest`
- 既定案や慣習から外す場合は、要件との対応理由を明記する
- 採用スタックが未確定なら、先に機能要件と非機能要件を固める

## 4. 設計方針
- 変更容易性を優先し、過度に早い抽象化を避ける。
- 外部依存は必要性が説明できるものだけを採用する。
- UI、API、データモデルなどの責務境界を曖昧にしない。
- 重要な既定値、制約、対象外はこの文書に残す。

## 5. 非機能要件
- 保守しやすい構成であること
- セキュアな既定値を持つこと
- ローカル開発と CI の再現性があること
- 対象実行環境で安定動作すること

## 6. 受け入れ条件
- `docs/initial-brief.md` の未確定事項が整理されている
- 開発コマンドとセットアップ手順が `README.md` に明記されている
- `lint / test / build` の実行方法が定義されている
- 重要ロジックに最低限のテスト方針がある
- 非目的が明文化され、実装範囲の逸脱を判断できる
- 依存追加や外部通信の必要性を説明できる

## 7. リリース前チェック
- 機密情報が含まれていない
- 不要な生成物やローカル専用ファイルが混ざっていない
- `README.md` と `docs/rules-coding.md` と実装の整合が取れている
- 既知の制約や未対応事項が必要に応じて記録されている
