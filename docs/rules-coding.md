# コーディングルール（人間向け）

この文書は実装判断の基準です。仕様は `docs/project-spec.md`、セットアップと運用は `README.md` を参照してください。

## 基本方針
- 変更は最小差分で行い、影響範囲を明確にする。
- 可読性・保守性・セキュリティを優先する。
- 仕様判断は `docs/project-spec.md` を優先し、日常運用は `README.md` に合わせる。
- 実装と仕様が衝突した場合は、この文書と `docs/project-spec.md` のどちらを更新すべきか整理してから直す。
- 技術固有のルールは、採用スタックが決まってから必要最小限で追加する。
- 採用した製品、プラットフォーム、フレームワークに公式または広く受け入れられたベストプラクティスがある場合は、正当な理由がない限りそれに沿って設計・実装する。

## 設計と責務
- 複数箇所で共有される責務は、正本となる振る舞いを一つ定義し、一貫して扱う。
- プラットフォーム差分や外部連携差分は適応層で吸収し、中心ロジックを分岐させない。
- 入出力、外部依存、ドメインロジックを分離し、中心ロジックを環境依存にしない。
- 同じ意味の値、判定、設定値、既定値、feature flag は散在させず、定義箇所を集約する。
- 分離や共通化は、責務境界と読みやすさが良くなるときに行い、まだ一箇所でしか使わず見通しもよい処理は無理に別ファイル化しない。
- 例外ケースを足す前に、責務分割やデータ構造の見直しで吸収できないかを先に検討する。
- 一時対応や互換処理を入れる場合は、削除条件や見直し時期を明示する。

## 実装ルール
- 処理の分割は、意図や境界が明確になる単位で行う。一連の流れを追う方が読みやすい処理や、単発で文脈依存が強い短い処理は呼び出し元の近くに置く。
- エラーは握りつぶさず、ログまたは返却値で明示する。
- 外部入力はバリデーションを行う。
- 命名は一貫性を優先する（英語推奨、必要に応じて日本語コメント）。
- 生成物やビルド成果物は直接編集しない。

## テスト/品質
- 重要ロジックにはテストを追加する。
- ビルドやテストが導入済みなら、変更時は最低限それらを実行してから共有する。
- 文言修正、表現の明確化、コメント更新など、振る舞いに影響しない変更では、導入済みのビルドとテストを省略してよい。
- CI を導入したら、可能な範囲で `lint / test / build` を自動化する。
- テストランナーや coverage 設定は、採用スタックに合わせて定義する。

## セキュリティ
- 機密情報をリポジトリに保存しない（`.env`, キー, 秘密鍵など）。
- 秘密情報をコードに埋め込まない。
- 依存更新時は脆弱性情報を確認する。
- 外部通信先や権限追加はレビュー対象にする。
- 外部通信や外部サービス追加時は、必要性と影響範囲を明記する。

## 技術固有ルールの追加
- 言語、ランタイム、フレームワーク固有のルールは、採用後に必要なものだけ追加・更新する。（例: 依存のインストールは `npm ci` や `bun install --frozen-lockfile` を使い、ローカルと CI の再現性を揃えるなど）
- 使わない技術の節は増やさず、不要なら書かない。

## 運用
- 公開用の成果物が必要な場合は、ローカル生成物ではなく CI で作成・配布する。
- ローカル専用の生成物、一時ファイル、検証用ファイル、個人用補助スクリプトは、明示的な理由がない限り含めない。
